# AddressSanitizer leak suppressions
# These are known leaks in third-party libraries, not in Vestige code.

# GLFW platform-init allocates X11 / GLX / EGL state whose lifetime equals
# the process — never explicitly freed. _glfwInitGLX, extensionSupportedGLX,
# and _glfwConnectX11 are all sub-paths of glfwInit(); a single suppression
# at the public entry point covers every backend.
leak:glfwInit

# Fontconfig allocates caches during GLFW/X11 initialization
# that are never freed. This is a known system library behavior.
leak:libfontconfig

# Mesa llvmpipe (the software rasterizer used in CI) lazily allocates
# internal pipe-state and pixel-buffer structures on first GL call. These
# come from a separately-loaded DSO that LSan cannot symbolize (frames
# show `<unknown module>`), so we match at the next named frame up — our
# test-only GL-parity helper. Real driver allocations on AMD/NVIDIA
# hardware fall under the same suppression with no false negatives, since
# the scope is test-helper-only (production code never calls in here).
leak:Vestige::Test::ShaderProgram

# libwayland-client allocates connection state on first wl_display_connect
# that persists for the process lifetime. Shows up under GLFW backends on
# Wayland sessions (parametric Photosensitive / Reference-case tests run
# this code path because GLTestEnvironment initialises GLFW at suite
# startup). Same lifetime-equals-process pattern as libfontconfig above.
leak:libwayland-client

# GLTestEnvironment::SetUp creates a hidden GLFW window + GL context at
# suite startup; the Mesa driver / wayland-egl shared object loaded by
# glfwCreateWindow allocates state that LSan reports under <unknown
# module> (DSO loaded after LSan's symbolizer cache). Lifetime equals
# process — the env's TearDown runs after LSan's check, so the leak is
# unavoidable. Suppress at the glfwCreateWindow entry point so the
# scope is bounded to the test environment, not production code.
leak:glfwCreateWindow
